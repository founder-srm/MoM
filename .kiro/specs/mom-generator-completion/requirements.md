# Requirements Document

## Introduction

This document specifies the requirements to complete the MoM (Minutes of Meeting) Generator web application. The UI foundation is fully built and working with mock data across all phases. The scope of this feature is to replace all mock/stub implementations with real backend integrations, fix identified bugs, and add missing functionality — resulting in a fully functional, production-ready application.

The system uses Next.js 16 App Router, Supabase (auth + Postgres + Storage), and the Groq API (Whisper for speech-to-text, Llama 3.3 for AI summarization). All UI components must continue to use shadcn/ui per project constraints.

**Current state summary (from codebase analysis):**
- All pages navigate to hardcoded `/meetings/draft-1/…` paths instead of real UUIDs
- All pages render mock data (`MOCK_MEETINGS`, `MOCK_TRANSCRIPT`, `MOCK_MOM`) instead of real data
- Login/signup uses `localStorage` instead of Supabase Auth
- The `useAudioRecorder` hook has a timer only — no real `MediaRecorder` integration and no `audioBlob` output
- `MeetingContext` has no `meetingId` field, making UUID propagation through the wizard impossible
- `groq-sdk` is not installed; no `/api/transcribe` or `/api/summarize` routes exist
- No `/api/auth/callback` route exists
- The `meetings` table schema is missing a `participants` column
- `font-sora` class is present in three files but Sora font is not loaded (should be `font-sans`)
- The `PATCH /api/transcripts` route expects `transcript_id` but the requirement uses `meeting_id` as the key

## Glossary

- **System**: The MoM Generator web application as a whole
- **Auth_Module**: The Supabase-based authentication subsystem (`@supabase/ssr` + `@supabase/supabase-js`)
- **Supabase_Client**: The browser Supabase client instantiated via `createBrowserClient` in `src/lib/supabase/client.ts`
- **Middleware**: The Next.js middleware at `src/middleware.ts` that handles session management and route protection
- **Dashboard**: The `/dashboard` page that lists all of the authenticated user's meetings
- **Meeting_Flow**: The multi-step wizard spanning New Meeting → Record/Upload → Transcript → MoM pages
- **Meeting_Context**: The React context (`MeetingProvider` / `useMeetingFlow`) that holds in-flight meeting state across the wizard pages; it must expose a `meetingId` string field for UUID propagation
- **Meeting_ID**: A UUID primary key generated by Supabase for each meeting record in the `meetings` table
- **Audio_Recorder**: The `useAudioRecorder` hook and its associated recording UI in `src/app/(app)/meetings/[id]/record/page.tsx`
- **Groq_Client**: The Groq SDK client initialized in `src/lib/groq.ts`
- **Transcription_API**: The Next.js API route at `/api/transcribe` (to be created) that calls Groq Whisper
- **Summarization_API**: The Next.js API route at `/api/summarize` (to be created) that calls Groq LLM
- **MoM_Renderer**: The `MomRenderer` component at `src/components/meeting/mom-renderer.tsx` that renders markdown-formatted MoM content
- **Participants_Field**: The meeting form field that holds a comma-separated string of participant names; must be persisted in the `meetings` table `participants` column
- **font-sora**: A CSS class currently referenced in `app-sidebar.tsx`, `(app)/layout.tsx`, and `mom-renderer.tsx` but never loaded (Sora font is not in the project); must be replaced with `font-sans` (Inter) in each file
- **Draft_Path**: The hardcoded `/meetings/draft-1/…` path currently used throughout the wizard, which must be replaced with a real Meeting_ID
- **Supabase_Storage**: The `meeting-audio` bucket in Supabase used to store uploaded and recorded audio files

## Requirements

### Requirement 1: Authentication Integration

**User Story:** As a user, I want to sign in and sign up using my email and password via Supabase, so that my meeting data is securely stored and private.

#### Acceptance Criteria

1. WHEN a user submits the login form with a valid email and password, THE Auth_Module SHALL authenticate the user via `supabase.auth.signInWithPassword` and redirect to `/dashboard` within 3 seconds.
2. WHEN a user submits the login form with invalid credentials, THE Auth_Module SHALL display an inline error message (e.g., "Invalid email or password") without navigating away from the login page.
3. WHEN a user submits the signup form with a valid email and a password of at least 8 characters, THE Auth_Module SHALL create a new account via `supabase.auth.signUp` and redirect to `/dashboard`.
4. WHEN a user submits the signup form with an email that is already registered, THE Auth_Module SHALL display an inline error message indicating the email is already in use.
5. WHEN an authenticated user clicks "Sign out", THE Auth_Module SHALL call `supabase.auth.signOut`, terminate the session (all session cookies removed), and redirect the user to `/login`.
6. THE System SHALL have no remaining calls to `localStorage.setItem("userEmail", …)`, `localStorage.getItem("userEmail")`, or `localStorage.removeItem("userEmail")` in the login page, signup form, `app-sidebar.tsx`, or any auth-related component.
7. WHEN an OAuth or email confirmation callback arrives at `/api/auth/callback` with a valid `code` query parameter, THE System SHALL exchange the code for a session via `supabase.auth.exchangeCodeForSession` and redirect to `/dashboard`.
8. WHEN the callback handler at `/api/auth/callback` receives an invalid or missing `code` parameter, THE System SHALL redirect to `/login?error=auth_failed`.
9. WHEN a user submits the signup or login form, THE System SHALL disable the submit button and show a loading indicator until the auth response is received, preventing duplicate submissions.

### Requirement 2: Route Protection via Middleware

**User Story:** As a system administrator, I want unauthenticated users to be blocked from protected app routes, so that meeting data is never exposed without a valid session.

#### Acceptance Criteria

1. WHEN an unauthenticated user navigates to any path matching `/dashboard` or `/meetings/*`, THE Middleware SHALL issue a redirect response to `/login`.
2. WHEN an authenticated user navigates to `/login` or `/signup`, THE Middleware SHALL issue a redirect response to `/dashboard`.
3. WHILE a user session exists and has not passed its expiry timestamp, THE Middleware SHALL call `supabase.auth.getUser()` to refresh the session token on each matched request.
4. WHEN a session token is expired or invalid, THE Middleware SHALL redirect the user to `/login` rather than passing through to the protected route.
5. THE Middleware SHALL NOT redirect requests whose paths start with `/api/auth/callback`, `/_next/static`, `/_next/image`, `/favicon.ico`, or that end in `.svg`, `.png`, `.jpg`, `.jpeg`, `.gif`, or `.webp`.

### Requirement 3: Meeting_Context UUID Propagation

**User Story:** As a developer, I want the Meeting_Context to store and propagate the real Meeting_ID UUID across wizard steps, so that each page can reference the correct database record without additional API calls.

#### Acceptance Criteria

1. THE Meeting_Context SHALL expose a `meetingId` field (typed as `string | null`, initially `null`) alongside the existing `meetingForm` and `meetingData` fields.
2. WHEN the new meeting form successfully creates a meeting record, THE System SHALL call a context setter to store the returned UUID in `Meeting_Context.meetingId` before navigating to the next step.
3. WHILE `Meeting_Context.meetingId` is non-null, THE record page, transcript page, and MoM page SHALL read it from context to construct API calls and navigation paths; none of those pages SHALL use hardcoded path segments such as `draft-1`.
4. WHEN the user navigates back from the transcript page to the record page, THE System SHALL use `Meeting_Context.meetingId` to construct the `/meetings/{meetingId}/record` path.
5. WHEN the user navigates back from the MoM page to the transcript page, THE System SHALL use `Meeting_Context.meetingId` to construct the `/meetings/{meetingId}/transcript` path.

### Requirement 4: Real Meeting CRUD via Supabase

**User Story:** As a user, I want my meetings to be created and retrieved from the Supabase database, so that my data persists across sessions and devices.

#### Acceptance Criteria

1. WHEN a user completes the new meeting form and clicks "Continue to Audio", THE System SHALL call `POST /api/meetings` with `title`, `date`, `participants`, and `description` fields; on success, THE System SHALL receive a real Meeting_ID UUID and store it in Meeting_Context before navigating forward.
2. WHEN the meeting record is created, THE Meeting_Flow SHALL navigate to `/meetings/{meeting_id}/record` using the real Meeting_ID instead of the Draft_Path.
3. WHEN the Dashboard page loads, THE Dashboard SHALL call `GET /api/meetings` to fetch the authenticated user's meetings ordered by `created_at` descending and render them; IF no meetings exist, THE Dashboard SHALL display an empty state with a "Create your first meeting" call-to-action button.
4. WHEN a meeting card is clicked on the Dashboard, THE System SHALL navigate to `/meetings/{meeting_id}` using the real UUID string.
5. WHEN the meeting detail page (`/meetings/[id]`) loads and the meeting exists but its MoM or summary has not yet been generated, THE System SHALL display the meeting title and status without error, showing placeholder text for the missing MoM/summary sections.
6. IF the meeting detail page (`/meetings/[id]`) loads and no meeting record is found for the given `id`, THE System SHALL display a "Meeting not found" message and a link back to the Dashboard.
7. WHEN a meeting is created via `POST /api/meetings`, THE System SHALL set the initial `status` field to `"created"` in the `meetings` table (not `"processing"` as currently implemented).
8. IF the `POST /api/meetings` call fails (network error or validation error), THE System SHALL display an inline error message on the new meeting form and SHALL NOT navigate away from the form.
9. THE `meetings` table (and the `POST /api/meetings` route) SHALL include a `participants` column of type `text` to persist the comma-separated participant names entered in the new meeting form.

### Requirement 5: Real Audio Recording via MediaRecorder

**User Story:** As a user, I want to record live meeting audio in my browser, so that I can capture a meeting without uploading a pre-recorded file.

#### Acceptance Criteria

1. WHEN a user selects "Record Live" and clicks the record button, THE Audio_Recorder SHALL request microphone access via `navigator.mediaDevices.getUserMedia({ audio: true })` before starting recording.
2. IF the user denies microphone permission, THE Audio_Recorder SHALL display an error message ("Microphone access is required. Click the lock icon in your browser address bar to allow it.") and show a "Try Again" button; the record button SHALL remain disabled until permission is granted.
3. IF the browser does not support audio recording (`MediaRecorder` or `getUserMedia` unavailable), THE Audio_Recorder SHALL display an error message ("Your browser does not support audio recording. Please use Chrome or Firefox, or upload an audio file instead.") and disable the record button.
4. WHEN recording is active, THE Audio_Recorder SHALL collect audio data chunks from a `MediaRecorder` instance at intervals not exceeding 1000ms.
5. WHEN recording is stopped, THE `useAudioRecorder` hook SHALL set `audioBlob` to a `Blob` containing all collected audio data in `audio/webm` format (or the best format supported by the browser's `MediaRecorder`).
6. THE `useAudioRecorder` hook SHALL expose `audioBlob` as `null` before recording starts and after `startRecording` is called until `stopRecording` is called.
7. THE `useAudioRecorder` hook SHALL expose `isRecording` as a boolean and `elapsed` as an integer count of whole seconds since recording started, capped at 7200 (2 hours).

### Requirement 6: Audio Upload to Supabase Storage

**User Story:** As a user, I want my recorded or uploaded audio to be saved to cloud storage, so that it can be retrieved for transcription without re-uploading.

#### Acceptance Criteria

1. WHEN a user clicks "Process Audio", THE System SHALL upload the audio to the `meeting-audio` Supabase Storage bucket under the path `{user_id}/{meeting_id}/audio.{ext}`, where `{ext}` is `webm` for recorded audio or the original file extension for uploaded files.
2. WHEN the upload is complete, THE System SHALL call `POST /api/audio` with `meeting_id`, `storage_url`, `file_name`, and `file_size` to save the audio file record to the `audio_files` table.
3. WHEN an upload file is selected, THE System SHALL validate that the file's MIME type or extension matches one of `mp3`, `wav`, `m4a`, `webm`, or `ogg`, AND that the file size is ≤ 50 MB; IF either check fails, THE System SHALL display an inline error message specifying the failed condition and SHALL NOT proceed with the upload.
4. WHILE the audio is uploading, THE System SHALL display a progress indicator showing the percentage of bytes uploaded (0–100%).
5. WHEN the audio upload and audio record save are both complete, THE System SHALL navigate to `/meetings/{meeting_id}/transcript` using the real Meeting_ID from Meeting_Context.
6. IF the Supabase Storage upload fails, THE System SHALL display an inline error message and provide a "Retry" button; THE System SHALL NOT navigate away from the audio page.
7. IF the `POST /api/audio` record save fails after a successful Storage upload, THE System SHALL display an inline error message indicating that the audio was uploaded but the record could not be saved, and SHALL provide a "Retry" button.

### Requirement 7: Groq SDK Installation and Client Setup

**User Story:** As a developer, I want the Groq SDK installed and a shared client configured, so that both transcription and AI summarization can use it.

#### Acceptance Criteria

1. THE System SHALL include `groq-sdk` as a listed production dependency in `package.json` with a pinned or tilde version (e.g., `"groq-sdk": "~0.9.0"`); as of the current codebase audit, this package is not yet installed.
2. THE System SHALL have a `src/lib/groq.ts` module that exports a singleton `Groq` client instance; IF `GROQ_API_KEY` is absent or an empty string at module load time, THE module SHALL throw an error with the message `"GROQ_API_KEY environment variable is not set"`.
3. THE `.env.local.example` file SHALL include a `GROQ_API_KEY=` entry alongside the existing Supabase variable entries.

### Requirement 8: Real Transcription via Groq Whisper

**User Story:** As a user, I want my meeting audio to be automatically transcribed to text, so that I can review and correct it before generating the MoM.

#### Acceptance Criteria

1. THE System SHALL have a `POST /api/transcribe` route (a new file, separate from the existing `/api/transcripts` CRUD route) that accepts a JSON body with a `meeting_id` field.
2. WHEN `POST /api/transcribe` is called with a missing or non-string `meeting_id`, THE Transcription_API SHALL return a 400 response with a JSON body containing an `error` field describing the validation failure.
3. WHEN `POST /api/transcribe` is called with a valid `meeting_id` that exists in both the `meetings` and `audio_files` tables, THE Transcription_API SHALL download the audio from Supabase Storage and send it to the Groq Whisper model (`whisper-large-v3`) within a 30-second timeout window.
4. WHEN transcription succeeds, THE Transcription_API SHALL upsert the transcript text into the `transcripts` table (inserting if no row exists for `meeting_id`, overwriting `transcript_text` if a row already exists) and update the meeting `status` to `"transcribed"`.
5. WHEN the transcript page loads for a meeting, THE System SHALL call `POST /api/transcribe` and display a progress indicator that cycles through the labels "Uploading", "Transcribing", and "Processing" until the API responds.
6. WHEN the Transcription_API returns a successful response, THE System SHALL display the returned transcript text in the editable `Textarea`, replacing any mock or placeholder content.
7. IF the Groq Whisper API call fails or times out, THE Transcription_API SHALL return a 502 response and THE System SHALL display an error message with a "Retry" button that re-invokes `POST /api/transcribe` for the same `meeting_id`.
8. WHEN `POST /api/transcribe` is called with a `meeting_id` that does not exist in the `audio_files` table, THE Transcription_API SHALL return a 404 response with `{ "error": "Audio file not found for this meeting" }`.
9. WHEN the user clicks "Generate MoM", THE System SHALL call `PATCH /api/transcripts` with `{ meeting_id, edited_text }` (using `meeting_id` as the lookup key, not `transcript_id`) to save the current textarea content; IF the PATCH call fails, THE System SHALL display an inline error and SHALL NOT navigate to the MoM page.

### Requirement 9: Real AI Summary and MoM Generation via Groq LLM

**User Story:** As a user, I want an AI-generated meeting summary and structured MoM produced from my transcript, so that I don't have to write meeting notes manually.

#### Acceptance Criteria

1. THE System SHALL have a `POST /api/summarize` route (a new file, separate from the existing `/api/summaries` CRUD route) that accepts a JSON body with a `meeting_id` field.
2. WHEN `POST /api/summarize` is called with a missing or non-string `meeting_id`, THE Summarization_API SHALL return a 400 response with a JSON body containing an `error` field.
3. WHEN `POST /api/summarize` is called with a valid `meeting_id`, THE Summarization_API SHALL use `edited_text` from the `transcripts` table if it is non-null and non-empty; otherwise THE Summarization_API SHALL use `transcript_text`; the response SHALL include a `transcript_version` field set to either `"edited"` or `"original"` indicating which was used.
4. WHEN the transcript and meeting metadata are fetched, THE Summarization_API SHALL send a single prompt to the Groq model `llama-3.3-70b-versatile` requesting a JSON response with two fields: `summary` (plain text) and `mom` (Markdown-formatted document with sections: Meeting Title, Date, Attendees, Agenda, Discussion Points, Decisions Made, Action Items as a Markdown table with Owner/Task/Deadline columns, and Next Steps).
5. WHEN the LLM response is received and parsed, THE Summarization_API SHALL upsert `summary` to the `summaries` table, upsert `mom` to the `moms` table, update the meeting `status` to `"completed"`, and return `{ summary, mom, transcript_version }` with HTTP 200.
6. WHEN the MoM page loads, THE System SHALL call `POST /api/summarize` and display a progress indicator showing named generation states ("Fetching transcript", "Generating with AI", "Saving results") until the API responds.
7. WHEN the Summarization_API returns successfully, THE System SHALL render the `mom` field via `MomRenderer` and display the `summary` field in the AI Summary tab.
8. WHEN the user clicks "Save to Archive" on the MoM page, THE System SHALL navigate to `/dashboard`; no additional save API call is needed since the MoM was already persisted in step 5.
9. IF the Groq LLM call fails or returns a non-parseable JSON response, THE Summarization_API SHALL return a 502 response and THE System SHALL display an error message with a "Retry" button that re-invokes `POST /api/summarize` for the same `meeting_id`.
10. IF `POST /api/summarize` is called with a `meeting_id` that has no transcript record, THE Summarization_API SHALL return a 404 response with `{ "error": "Transcript not found for this meeting" }`.

### Requirement 10: Export Features

**User Story:** As a user, I want to export my MoM in multiple formats, so that I can share it easily with meeting participants.

#### Acceptance Criteria

1. WHEN a user clicks "Copy as Markdown", THE System SHALL write the raw Markdown text (preserving `**bold**`, `---` dividers, `|` table syntax, `#` headings, `>` blockquotes, `-` list items, and backtick code spans) to the clipboard; WHEN the write succeeds, THE System SHALL display a "Copied!" confirmation label on the button for 2 seconds.
2. IF the clipboard write fails for "Copy as Markdown", THE System SHALL display an inline error message ("Could not copy to clipboard. Please copy manually.") adjacent to the button.
3. WHEN a user clicks "Print / Save as PDF", THE System SHALL call `window.print()` to open the browser's native print dialog.
4. WHEN a user clicks "Copy MoM" (plain text), THE System SHALL write the MoM text stripped of Markdown syntax characters (`**`, `#`, `>`, `-`, `*`, `|`, `` ` ``, `---`) to the clipboard; WHEN the write succeeds, THE System SHALL display a "Copied!" confirmation label on the button for 2 seconds.
5. IF the clipboard write fails for "Copy MoM" (plain text), THE System SHALL display an inline error message adjacent to the button.
6. WHILE on the MoM page or meeting detail page, THE System SHALL display all three export actions — "Copy MoM" (plain text), "Copy as Markdown", and "Print" — as distinct, labelled buttons.

THE `PATCH /api/transcripts` route SHALL accept `meeting_id` as the lookup key (resolving `transcript_id` for the where-clause internally) to match the calling convention described in Requirement 8, criterion 9; the current implementation that requires the client to supply `transcript_id` directly SHALL be updated.
10. WHEN `POST /api/meetings` creates a meeting, THE System SHALL insert `status: "created"` (not `status: "processing"` as currently implemented) to align with the status lifecycle defined in the planning documents.

## Future Scope

### Cloud Migration of the Local AI Pipeline

The current AI pipeline (Faster-Whisper + Ollama) is intentionally local-first to eliminate API costs and keep all data on-premises. When scale or reliability demands change, the pipeline can be migrated to a dedicated cloud server with minimal impact on the rest of the system.

**Migration path:**

1. **Faster-Whisper → hosted transcription endpoint**: Deploy the existing `services/transcription` FastAPI service to a cloud VM or container (e.g., AWS EC2, GCP Compute Engine, or a GPU-enabled container service). Update the `TRANSCRIPTION_SERVICE_URL` environment variable to point to the new host. No Next.js code changes are required.

2. **Ollama → cloud inference server**: Deploy an Ollama instance (or an OpenAI-compatible inference layer such as vLLM) on a cloud GPU instance. Update the `OLLAMA_BASE_URL` environment variable to point to the new host. No Next.js code changes are required.

3. **Frontend architecture remains unchanged**: Next.js, Supabase Auth, Supabase PostgreSQL, Supabase Storage, shadcn/ui, and Tailwind CSS are unaffected by this migration. Only the two environment variables and the infrastructure hosting the AI services change.

4. **Authentication and rate-limiting**: When moving to a shared cloud server, the transcription and Ollama endpoints should be placed behind an API gateway or reverse proxy with token-based authentication and per-user rate limiting to prevent abuse.

This design ensures a clean separation between the frontend application and the AI inference layer, making the migration entirely an infrastructure concern rather than a code concern.
